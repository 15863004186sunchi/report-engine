# 部门报表数据引擎 - 详细设计文档

## 1. 系统架构概览

本项目采用离线计算与实时查询相结合的架构，以应对复杂报表查询带来的数据库性能瓶颈。整体架构分为数据采集录入层、ETL计算聚合层、数据存储层和数据服务呈现层。

*   **前端展示层**：Vue3 + ECharts，负责各种维度BI看板的渲染。
*   **后端服务层**：Spring Boot/Java，提供RESTful API接口，供前端查询聚合后的指标数据。
*   **计算调度层**：XXL-JOB 定时任务框架，调度夜间批处理任务。
*   **数据存储层**：MySQL 数据库环境，拆分为业务明细库（OLTP）和 报表宽表库（OLAP/Data Mart）。

## 2. 核心模块与数据流转设计

### 2.1 数据流转时序
1.  **日常业务流**：各业务系统（工时系统、报销系统、项目管理系统）日常将基础数据写入MySQL业务明细表。
2.  **人工辅助录入**：运营人员通过系统后台页面/Excel导入，将“Plan数据、GAAP预估数据、目标毛利率”等预估绩效目标录入MySQL配置表。
3.  **定时ETL计算（核心侧重）**：
    *   **触发时机**：XXL-JOB 每日凌晨触发（如 01:00）。
    *   **计算逻辑**：Java定时任务代码读取明细表及配置表，按照预定的业务公式（见需求文档），在内存或通过SQL聚合出项目维度及部门维度的汇总数据。
    *   **数据落地**：将最终计算的指标数据（月度、年度ITD等）`Upsert` (插入或更新) 到专门的“报表聚合宽表”中。
4.  **报表前端查阅**：前端Vue发起请求，后端API直接查询预先计算好的“报表聚合宽表”，极速返回 JSON 供 ECharts 渲染。

### 2.2 数据库设计预案 (MySQL)

为了支撑高性能报表查询及ETL，我们需要设计以下几类表模型：

#### 2.2.1 基础维度/配置表 (Dimension/Configuration)
*   `sys_department`：部门字典表。
*   `sys_project`：项目信息表（包含主键、项目名称、项目属性类型[交付/售前/内部/运营]、税率等）。
*   `rep_manual_target`：运营手工录入目标表。
    *   字段：ID, 部门ID, 年份, 月份, 预估GAAP收入, GAAP收入目标, 目标毛利率, 本年度已签约合同额, 本年度签约绩效目标、调整后的PLAN各项成本等。

#### 2.2.2 基础明细流水表 (Fact - 业务端提供，按需抽取)
*   `biz_timesheet_detail`：人员工时及费率明细流水。
*   `biz_expense_detail`：报销及其他运营成本流水。
*   `biz_invoice_detail`：项目开票明细。

#### 2.2.3 核心：报表聚合宽表 (Report Mart Tables)
这是ETL产出的最终表，前端API直接查这些表。

**A. 表名：`rep_project_monthly_summary` (项目月度汇总宽表)**
*   **主键聚合键**：`project_id`, `year`, `month` (建唯一联合索引)。
*   **指标字段**：开票收入, 不含税开票收入, GAAP收入达成, 正式人力成本实际, 外包人力成本实际, 运营成本实际, 报销费用实际, 预估后续正式人力, 预估后续外包人力等...
*   **累计指标 (ITD)**：当年 ITD 开票收入, 去年同期 ITD 开票收入... (为提升查询速度，可将ITD等计算后的累计值随月度记录一起固化)。

**B. 表名：`rep_department_monthly_summary` (部门月度核心指标宽表)**
该表是对项目表数据的按门类及计算公式二次聚合汇总，直接对接图2的UI展示。
*   **主键聚合键**：`dept_id`, `year`, `month` (建唯一联合索引)。
*   **指标字段**：
    *   收入类：部门总开票收入, 部门不含税开票收入, 部门GAAP收入达成。
    *   成本类：部门实施成本（交付类）, 部门售前成本（售前类）, 部门其他成本（内部/运营类）。
    *   利润率公式结果：毛利率A_开票, 毛利率A_GAAP, 毛利率B_开票, 毛利率B_GAAP。
    *   当年累计ITD类字段：本年ITD_实施成本, 本年ITD_售前成本, 毛利率A_ITD...
    *   去年同期ITD类字段及同比数据等。

## 3. Java后端 (报表数据处理架构) 指导

为了实现“高复用性、高性能的业务数据聚合代码”，建议采用如下代码分层和设计模式：

### 3.1 提取/转换/加载 (ETL) 任务模块
*   `DataLoaderService`：抽象接口，定义从不同业务微服务/数据库拉取明细数据的方法（避免后续数据源变动影响核算代码）。
*   `FormulaCalculator`：核心策略类，利用策略模式(Strategy Pattern)封装各项财务指标的计算公式。
    *   比如提供单独的 `calcGrossMarginA()`, `calcITD()` 方法，使得公式易于单元测试维护。
*   `EtlJobHandler` (基于 XXL-JOB 的注解)：调用 DataLoader 拉取前一日所有变动数据 -> 传入 Calculator 计算各类成本/利润率 -> 批量调用 MyBatis-Plus 的 `saveOrUpdateBatch` 写入宽表。

### 3.2 报表查询API模块
*   `ReportController`：提供标准化的 RESTful 数据接口给 Vue3 端。
*   设计专门的 JSON Response 结构，适配 ECharts 需要的数据格式（如 X 轴分类数组 
`xAxisData:['1月','2月'...]`, Y 轴图例序列 `series:[{name:'预收入', data:[...]}]`）。通过后端组装好结构层级，减少前端 Vue 组件的数据清洗压力。

## 4. UI与交互接口映射建议 (对照 图2)

根据UI图划分如下关键接口提供给前端：
1.  **`/api/report/overview/header`**：部门维度顶部看板数据 (已签合同额, 绩效目标, 售前成本使用率, 部门总ITD收入)。
2.  **`/api/report/trend/monthly`**：中间月度柱状图趋势数据。入参：年份、部门ID、指标类型(如:GAAP收入、毛利率A等)。返回 ECharts 所需格式和右侧明细表格 List。
3.  **`/api/report/compare/departments`**：底部多部门横向折线图比对数据。入参：年份、指标枚举。返回各部门按月的折线序列数组。

## 5. 项目潜在技术难点与规避建议
*   **数据订正/历史重算**：当由于漏报导致跨越多个月修改前序月份的人员工时信息时，ETL任务需具备“区域重算 (Recalculate)” 的容错机制，而不仅仅是增量计算昨天。建议设计**指定时间区间的全量重洗定时任务**。
*   **公式链式依赖**：如EAC依赖ETC和ITD，部门其它成本依赖个人明细数据再做比例分摊。在ETL调度顺序上必须严格基于依赖图(DAG)顺序或在代码的内存对象内保证按拓扑排序进行结算。
