# 部门报表数据引擎 - Codex / AI 辅助开发指导指南

这份文档专为 AI Coding Assistant (如 Codex/Cursor/Antigravity 等) 编写。当你在本项目 (`E:\antigravityPro\report-engine`) 下执行代码开发任务时，请严格遵循此文档的架构约束、技术栈规范和业务计算逻辑。

## 1. 工程基础规范

*   **技术栈约定**：
    *   **后端**：Java 17 或以上, Spring Boot 3.x, MyBatis-Plus (用于数据库 ORM), MySQL 8.x, XXL-JOB (分部定时调度核心)。
    *   **前端**：Vue 3 (Composition API, `<script setup>`), TypeScript, ECharts 5.x。
*   **代码分层强制约定 (大忌：不要在Controller里手写复杂计算！)**：
    *   `Controller` 层：**绝对不含有任何财务计算公式逻辑**。只负责接收参数、校验、调用 Service，并返回针对 ECharts 视图结构化封装好的 JSON Result。
    *   `Service` 层：核心业务流转。
    *   `Calculator / Strategy` 引擎包：**所有的毛利、ITD、ETC 的加减乘除公式，必须单独提炼在此层**。这部分代码**必须**有独立、高覆盖率的单元测试 (JUnit 5 + Mockito)。
    *   `Mapper` 层：单纯的 CRUD，复杂连表查询可手写 XML。宽表的批量写入使用 MyBatis-Plus 提供的 `saveOrUpdateBatch(...)`。

## 2. 数据库基座与核心表定义

在你开始写业务代码前，请按照此结构生成 `schema.sql` 并创建对应的 Java Entity 类。

*   **原始流水数据表 (数据源/只读拉取阶段)**：
    假定已有表或通过接口封装：`biz_timesheet`（工时明细）、`biz_expense`（报销明细）、`biz_invoice`（开票明细）。
*   **配置运维表 (CRUD阶段)**：
    *   `sys_project` (id, name, type[0交付/1售前/2内部/3运营], tax_rate)
    *   `sys_department` (id, name, parent_id)
    *   `rep_manual_target` (年份, 月份, 部门ID, GAAP收入预估, GAAP收入目标, 目标毛利率, 签约绩效等，由运营通过页面维护)。
*   **ETL批处理结果宽表 (报表查询核心，聚合宽表)**：
    在编写向这些宽表写入的代码时，**主键约束/唯一索引 (如 `year`+`month`+`dept_id`) 至关重要**，必须防止定时任务重跑导致数据翻倍。
    *   `rep_project_monthly_summary` (以“项目+月份”为维度)
    *   `rep_department_monthly_summary` (以“部门+月份”为维度，直接对接最终前端各图表展示)

## 3. 开发阶段 1：构建计算引擎与 ETL 调度 (最核心任务)

当分配给你“实现毛利计算或成本汇总任务”时，你的思考路径应如下：

**步骤 A：定义公式接口**
参考《需求文档》，先不要急着连数据库，先把纯数学公式写成 Java 方法。比如：
```java
// 计算某部门上月的 毛利率 A
public BigDecimal calcGrossMarginA(BigDecimal implementCost, BigDecimal taxFreeInvoiceSum);
```
**步骤 B：实现 ETL Job 调度 (由 XXL-JOB 触发)**
1.  **Extract (拉取)**: 通过 MyBatis 聚合查询出前一天/该月所有变动底表数据。
2.  **Transform (转换与计算)**: 遍历查询出的原始对象，调用步骤 A 中的公式。注意：ITD代表累加值（本年迄今），你需要拉取年初至今的数据或者基于上月ITD+本月发生额进行计算。
3.  **Load (落盘)**: 将内存中组装完毕的 `RepProjectMonthlySummary` List 实体集合，执行 `myBatisPlusService.saveOrUpdateBatch(list, 1000)`。

*AI 开发避坑指南*：金额计算**必须且只能**使用 `java.math.BigDecimal`，设定好统一的舍入模式 (`RoundingMode.HALF_UP`) 和精度 (如 `setScale(2)`)，严禁使用 `Double`！

## 4. 开发阶段 2：报表 API 与出参封装 (为前端服务)

当分配给你“实现某某报表查询接口”时，你的目标是**让前端 Vue 不写业务逻辑，拿到数据直接塞给 ECharts**。

**不好的 API 返回格式**：
```json
[{"month":"1月","gaap": 100}, {"month":"2月","gaap":200}]
```
前端还需要遍历拆数组。

**强制要求的友好 API 封装格式 (供大屏图表使用)**：
```json
{
  "code": 200,
  "data": {
    "xAxis": ["1月", "2月", "3月"], // ECharts X轴需要的一维数组
    "series": [
      {
        "name": "GAAP预估收入",
        "type": "line",
        "data": [100, 200, 150]    // 严格与 X 轴对齐的 Y 轴数据
      },
      {
        "name": "GAAP实际收入",
        "type": "bar",
        "data": [90, 210, 140]
      }
    ]
  }
}
```
**提示**：在后端建立一层 `ChartResponseDTO` 对象，由后端的 `ReportService` 将宽表取出的行数据行列转化为列数据的组装格式。

## 5. 前端开发要点 (Vue3 + ECharts)

当需要在前端实现界面时：

1.  **模块化划分**：按图纸将其拆分为 `<TopCards />`, `<TrendChart />`, `<DepartmentCompareChart />` 三个独立组件。
2.  **网络层**：使用 Axios。建立独立的 `api/report.ts`。所有的请求应该在最顶层父级页面 (如 `ReportDashboard.vue`) 发起，获取到数据后，通过 Vue 的 `props` 单向流传递给底下的子图表组件。
3.  **ECharts 自适应**：注意在图表组件 `onMounted` 实例化 ECharts 时，务必监听浏览器 `resize` 事件或者使用 `ResizeObserver`，防止当屏幕缩放时出现变形。当组件 `onUnmounted` 时，一定要调用 `chartInstance.dispose()` 清理内存泄漏。

## 附录：处理复杂需求的思维提示

如果遇到某些需求中诸如“包含售前但是只看本部门的人所产生的费用”这种复合型查询：
1.  优先考虑是否可以在 ETL 夜间跑批的时候，就通过一条稍微复杂点的 SQL `GROUP BY` 取出并存入宽表对应字段。
2.  **绝对不要**在用户打开网页看报表的请求里，去即时联表JOIN千万级的工时明细表计算上述逻辑，这会导致查询接口立刻超时 (HTTP 504)。
3.  万一前一天数据报错了怎么办？你的存储层(`saveOrUpdate`)与ETL作业设计中，都要允许传入 `startDate` 和 `endDate`，以便手动跑批重算上周/上月的数据，这也是为什么宽表需要带上时间戳复合主键的原因。
